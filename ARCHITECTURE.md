# ARCHITECTURE.md — The Cloud School

> Onboarding guide for future developers. Explains how this project is structured,
> why it is built the way it is, and how to extend it without fighting the framework.

---

## 1. What this is

A marketing website for **The Cloud School** — an AI/Cloud/DevOps training
business in Pallikaranai, Chennai. The live site is currently a **single
landing page** (`/`) that scrolls through sections: hero, program, workshops,
roadmaps, trainer, testimonials, a consultation form, contact, and footer.

Despite being a one-pager, it runs on a **full-stack SSR meta-framework**
(TanStack Start). The framework is currently over-provisioned for the content,
but it is in place to (a) satisfy the Lovable integration and (b) leave room
to grow into multi-page routing, server functions, and data fetching.

**Stack at a glance**

| Concern     | Tech                                             |
| ----------- | ------------------------------------------------ |
| Framework   | TanStack Start (`@tanstack/react-start`)         |
| UI runtime  | React 19.2                                       |
| Routing     | TanStack Router 1.170 (file-based)               |
| Data        | TanStack Query 5.101 (initialized, not yet used) |
| Server      | Nitro → Cloudflare (default target)              |
| Build       | Vite 8 (via `@lovable.dev/vite-tanstack-config`) |
| Styling     | Tailwind CSS v4 (CSS-first config)               |
| UI kit      | shadcn/ui (New York) on Radix UI primitives      |
| Language    | TypeScript 5.8 (strict)                          |
| Package mgr | bun                                              |

## 3. Request / render flow

```
Browser request
      │
      ▼
src/server.ts  (Nitro/Cloudflare entry)
      │  wraps @tanstack/react-start/server-entry
      │  normalizes "swallowed" h3 SSR 500s into a friendly error page
      ▼
src/start.ts  (createStart)
      │  requestMiddleware: errorMiddleware (catches non-HTTPError throws)
      ▼
src/router.tsx  (createRouter)
      │  routeTree + QueryClient in context, scrollRestoration on
      ▼
src/routes/__root.tsx  (shell: <html>, <head>, <Outlet/>, <Scripts/>)
      │  provides QueryClientProvider, renders the matched route
      ▼
src/routes/index.tsx  (the page)
```

**Error handling has four cooperating layers** — understand them before
touching any of them:

1. `server.ts` — wraps Nitro's server entry. Nitro/h3 sometimes swallows
   in-handler throws into a generic JSON 500 (`{"unhandled":true,...}`).
   `server.ts` detects that body shape and replaces it with `renderErrorPage()`.
2. `error-capture.ts` — captures the _original_ Error out-of-band (via global
   `error`/`unhandledrejection` listeners) so `server.ts` can recover the real
   stack, because h3 has already consumed the throw.
3. `start.ts` — `errorMiddleware` re-throws HTTP-error responses untouched but
   wraps other server errors in the static error page.
4. `__root.tsx` — React-level `errorComponent` / `notFoundComponent` for
   client-side failures, forwarding to Lovable telemetry via
   `reportLovableError`.

## 4. Routing

- **File-based.** Every `.tsx` file in `src/routes/` is a route.
- Only one exists today: `index.tsx` → `/`.
- In-page navigation is **anchor-based** (`#program`, `#workshops`, …) with
  CSS `scroll-behavior: smooth`. There is no client-side route navigation yet.
- The root shell is **only** `__root.tsx`. Do not create `src/pages/`,
  `app/layout.tsx`, or `_app/` (those are Next.js/Remix conventions and will
  break the build).

See `src/routes/README.md` for the full convention table
(`$param`, `_layout.tsx`, `$.splat`, optional `{-$x}` segments).

## 5. Styling & design system

Defined entirely in `src/styles.css` (Tailwind v4 CSS-first config — there is
no `tailwind.config.js`; everything lives in `@theme` / `@utility` blocks).

**Color tokens (OKLCH color space), as CSS variables:**

| Token          | Role                           |
| -------------- | ------------------------------ |
| `--cream`      | warm off-white background base |
| `--ink`        | near-black foreground/text     |
| `--ember`      | **brand accent** (orange)      |
| `--ember-soft` | lighter ember for glows        |

Plus the full shadcn token set (`--background`, `--primary`, `--card`,
`--muted`, `--border`, `--ring`, …). A `.dark` theme is **defined but has no
toggle/provider** — it is currently unreachable.

**Fonts:** Fraunces (`--font-display`, serif headings) + Inter (`--font-sans`),
loaded from Google Fonts (with `preconnect`). Override via the
`font-display` / `font-sans` utilities.

**Custom utilities to reuse:**

- `glass`, `glass-dark` — frosted glassmorphism panels
- `grain` — subtle noise overlay
- `animate-float-slow`, `animate-float-med`, `animate-drift`,
  `animate-fade-up`, `animate-pulse-ring` — CSS keyframe animations
- `text-balance`, `text-pretty` — typographic wrapping

**Class merging:** always compose classes through `cn()` (`src/lib/utils.ts`),
which combines `clsx` + `tailwind-merge`. Do not concatenate class strings by hand.

## 6. State management

- **No global state library.** App state is local `useState` only:
  - nav `scrolled` state (`Nav`)
  - form `submitted` state (`Consultation`)
- TanStack Query's `QueryClient` is created in `router.tsx` and provided in
  `__root.tsx`, but **no queries/mutations are used yet**. It is ready for
  data fetching when you add it.

---

## 7. Forms

There is exactly **one form** — the consultation form in `index.tsx`
(`Consultation` component).

- Raw, **uncontrolled** `<input>` elements with the native `required` attribute.
- `onSubmit` calls `preventDefault()`, then flips a local `submitted` boolean
  to show a "Thanks — we'll be in touch" state.
- **It does not submit anywhere.** No network call, no email, no API. It is a
  visual placeholder.

The following are **installed but unused** and should back any real form work:
`react-hook-form`, `zod`, `@hookform/resolvers`.

---

## 8. UI component library — important inconsistency

`src/components/ui/` contains **47 shadcn/ui primitives** (Button, Card,
Dialog, Form, etc.) built on Radix UI. **None of them are imported by the
application.** `index.tsx` hand-rolls every visual element with raw Tailwind
classes, duplicating card/button styling inline.

> **Decision needed before extending the UI:** either
> **(a) adopt** the shadcn components (import `Button`, `Card`, `Input` from
> `@/components/ui` and standardize on them), or
> **(b) delete** `src/components/ui/` and the Radix dependencies.
>
> The current state — a full unused component library sitting next to
> hand-rolled equivalents — is the main inconsistency in the codebase.

The `Button` primitive is a good reference for the intended pattern: it uses
`class-variance-authority` (`cva`) for variants and `Slot` (`asChild`) for
composition.

## 9. Animations

Pure CSS — **no JS animation library** (no Framer Motion). Six keyframes live
in `styles.css` (`float-slow`, `float-med`, `drift`, `fade-up`, `pulse-ring`,
`shimmer`) and are exposed as `animate-*` utilities. The hero relies heavily on
`backdrop-blur` (glass) and large blurred radial-gradient "blobs".

## 11. Build, lint, deploy

**Package manager: bun.** `bunfig.toml` enforces a 24-hour minimum package
age for new dependencies (supply-chain guard); Lovable's own packages are
exempted.

| Script      | Command                         | Purpose                                                                             |
| ----------- | ------------------------------- | ----------------------------------------------------------------------------------- |
| `dev`       | `vite dev`                      | Dev server (Lovable wrapper injects devtools, Tailwind, tsconfig paths, Nitro, HMR) |
| `build`     | `vite build`                    | Production build → `.output` (Nitro/Cloudflare)                                     |
| `build:dev` | `vite build --mode development` | Dev-mode build                                                                      |
| `preview`   | `vite preview`                  | Preview the production build                                                        |
| `lint`      | `eslint .`                      | Lint (strict, Prettier-integrated)                                                  |
| `format`    | `prettier --write .`            | Format                                                                              |

**Lint rule worth knowing:** ESLint **forbids `server-only`** (the Next.js
package). Server-only code must use `*.server.ts` files or TanStack's
`@tanstack/react-start/server-only` marker.

**Deployment:** managed by **Lovable** and targets **Cloudflare** via Nitro.
Pushing commits to the connected branch syncs to Lovable and deploys — there is
no separate CI/CD config in this repo. There is **no** `wrangler.toml`,
Dockerfile, or GitHub Actions file.

> **`AGENTS.md` constraint:** do **not** rewrite already-pushed git history
> (force-push, rebase/amend/squash) — it desyncs Lovable and the user loses
> project history. Keep the branch in a working state.

## 12. How to add features (matches the existing architecture)

1. **New page** → create `src/routes/<name>.tsx` with
   `createFileRoute("/<name>")({ component })`. Nested layouts use
   `_layout.tsx`; dynamic segments use `$id`. It auto-registers in
   `routeTree.gen.ts`. Do **not** introduce React Router or a separate API
   server — TanStack Router/Start already provide these.

2. **UI consistency** → first resolve the shadcn decision in §8. Then either
   import from `@/components/ui` or keep raw Tailwind via extracted helpers.

3. **Reusable layout helper** → the `Section({ id, eyebrow, title, muted,
children })` function inside `index.tsx` is the canonical section wrapper.
   **Extract it to `src/components/Section.tsx`** and reuse it for every new
   section. Likewise `LogoMark` and `Field` should move into `src/components/`.

4. **Forms** → use `react-hook-form` + `zod` + `@hookform/resolvers` for
   validation, and submit through a **`createServerFn`** in a `*.server.ts`
   module so the form actually reaches a backend.

5. **Data** → use TanStack Query (client already in router context). Define
   route `loader`s for SSR-friendly data.

6. **Styling** → stay inside the Tailwind v4 + OKLCH token system. Use
   `font-display`/`font-sans`, the `--ember` accent, the `glass` utility, and
   the `animate-*` utilities. Add new keyframes as `@utility` blocks in
   `styles.css`.

7. **Server logic** → place it in `*.server.ts`; register cross-cutting
   middleware in `src/start.ts`. Keep the existing error-recovery layers intact
   unless you are deliberately simplifying them.

8. **SEO for new pages** → set per-route `head` meta in each route's
   `createFileRoute` options (overriding/extending `__root`); add
   `sitemap.xml` and JSON-LD `LocalBusiness`/`Course` blocks.

## 13. Known gaps (for planning)

- Consultation form has no backend or validation.
- Only one route — no About / Courses / Blog / Testimonials pages.
- Dark theme defined but no toggle/provider.
- No `tsc` typecheck step in the build pipeline (build is just `vite build`).
- `index.tsx` is ~870 lines with inline helpers that should be extracted.
- No tests, no analytics, no i18n (Tamil would help a Chennai audience).
- SEO assets missing: `robots.txt`, `sitemap.xml`, `og:image`, JSON-LD,
  canonical link.
- Unused dependencies inflate the install (see §10).
